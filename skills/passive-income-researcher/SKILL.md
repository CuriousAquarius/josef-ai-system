# Passive Income Researcher

## What "Passive Income" Actually Means (For a Beginner)

True passive income (money while you sleep with no ongoing work) is rare and usually comes after significant upfront work. Realistic goal: **semi-passive income** — build something once, maintain it lightly, and it generates revenue regularly.

Good AI-powered options for a beginner:
- A tool people pay a monthly fee to use (SaaS — Software as a Service)
- A service people pay per use (pay-as-you-go automation)
- Digital templates or configurations people buy once

## Step 1 — Run the Scoring Formula (ALWAYS, before anything else)

When Josef pitches any new idea, score it FIRST. Five factors, each rated 1–5, then multiplied together:

1. **Customer income** — can the target customer actually afford to pay?
2. **Usage frequency** — daily tool or once-a-year tool?
3. **Market gap** — how underserved is this need?
4. **Can't build themselves** — how hard is it for the customer to DIY?
5. **Stickiness** — once they start, how painful is it to leave?

**Verdict:** 500+ = build · 200–500 = consider · below 200 = skip

Show the per-factor scores and the product, then the verdict. Full details: `feedback_scoring_formula.md` in memory.

Also load for context: `Obsidian/Research/micro_saas_playbook.md` and `Obsidian/Pipeline/next_step_for_revenue.md`.

## Step 2 — The 5 Questions

Before building anything, answer all 5:

1. **Who pays for this?** (Be specific — not "businesses" but "solo freelancers who manage client emails")
2. **What problem does it solve?** (What do they currently do instead? How painful is it?)
3. **Why would they pay vs. use a free alternative?** (What's the unique value?)
4. **How much would they pay?** ($5/month? $50/month? One-time $20?)
5. **Can a beginner actually build this?** (Honest assessment of skill gap)

If you can't answer all 5 clearly → the idea needs more definition before proceeding.

## Beginner-Realistic AI Business Models (Josef's actual stack)

### Model 1: Chatbot-as-a-Service (already running — Serra)
Embed an AI chatbot on a client's website, charge monthly.
- Example: Serra — $79/mo Standard, $129/mo Pro
- What you build: per-client knowledge file + embed script (system already exists)
- New ideas here = new verticals for Serra, not new infrastructure
- Revenue potential: Medium-high, recurring

### Model 2: Productized Service (already running — Serra Studio)
Fixed-price service with a recurring management tier.
- Example: Serra Studio — $750 site + $129/mo Full Stack Management
- What you build: repeatable delivery process, not software
- Revenue potential: Medium-high — the monthly tier is where passive income lives

### Model 3: AI-Powered Tool (Simple Web App)
A website where people input something and AI does the work.
- Example: "Paste a job description → get a tailored cover letter"
- What you build: Simple web form → Claude API → output
- Skill required: Basic web development, Claude API (Josef has both)
- Revenue potential: Medium ($500–$5000/month at scale)

### Model 4: Content/Channel Plays (already running — lo-fi YouTube, USANA)
Automated content pipelines that compound over time.
- Example: lo-fi channel (24/7 stream + uploads), USANA Instagram (auto-post pipeline)
- What you build: generation + scheduling scripts; the channel grows while you sleep
- Revenue potential: Low at first, compounds — slowest to start, cheapest to run

## The MVP Principle

MVP = Minimum Viable Product. The simplest version that proves someone will pay.

**Rule:** Before building anything polished, find 3 people willing to pay for the ugly version.

If no one pays for the rough version → the idea needs to change, not get more polished.

## Red Flags in an Idea

- "Everyone would want this" → Too broad. Who specifically?
- "It's like X but better" → Why would people switch from X?
- "I just need to build it and they'll come" → Distribution is usually harder than building
- Requires skills the user doesn't have yet (e.g., complex backend infrastructure)
- Depends on a platform that could remove access (e.g., building on a closed API that could be revoked)
